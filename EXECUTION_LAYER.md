# Execution Layer Design

The execution layer is the part of Casper that actually touches AWS. Everything else in the system — policies, predictions, LLM proposals, UI — produces inputs for this layer or reads its output. Its primary output is **evidence**, not infrastructure change. The infra change is almost a side effect; the evidence is the product.

This document describes how the layer is designed and why.

---

## What "transparent execution" means

Transparency here is a concrete, four-part property:

1. **Pre-execution transparency** — before any AWS call fires, the system has stated *exactly* what it's about to do, in a form a human can verify against the proposal.
2. **In-flight transparency** — while the action runs, every step (API call, response, poll, decision) is observable in real time, not reconstructed afterward.
3. **Post-execution transparency** — after the action completes, fails, or rolls back, there is a complete causal record: every input, every API call, every response, every state transition, every decision, with timestamps and hashes.
4. **Reproducibility** — given only the audit log, anyone can answer *"why did this happen?"* without reading the source code.

The execution layer's job is to generate the evidence for all four. Transparency is not a logging concern bolted onto the side. It is the layer's reason to exist.

---

## Design principles

### 1. Plans, not code

When a proposal reaches the execution layer, the **first** thing that happens is the proposal is compiled into a typed `ExecutionPlan` — an ordered list of `Step` objects. Nothing executes until the full plan exists and is persisted.

A `Step` is roughly:

```
{
  id: "step-3",
  kind: "aws_api_call" | "poll" | "verify" | "decision" | "rollback_marker",
  description: "Resize orders-prod from db.r6g.large to db.r6g.xlarge",
  api: { service: "rds", operation: "ModifyDBInstance", params: {…} },
  preconditions: […],         // what must be true before this step runs
  postconditions: […],        // what must be true after for it to count as done
  timeout: "10m",
  on_failure: "rollback" | "abort" | "continue",
  rollback_step_id: "step-7"  // pointer to the inverse step
}
```

For RDS resize, the forward plan is roughly:
`describe-current-state → modify-instance → poll-until-modifying → poll-until-available → verify-success-criteria`.
The rollback plan is a parallel list:
`modify-instance-back → poll-until-available → verify-rolled-back`.

**Why this matters:** the plan is shown in the UI *before* approval. The user is not approving "resize the DB." They are approving an ordered list of API calls with concrete parameters. There is no hidden execution logic — what's in the plan is what runs.

### 2. One deterministic interpreter

Once approved, a single worker reads steps from the plan and executes them one at a time. The worker has **no per-action logic**. It knows how to:

- Mark a step `running`, persist
- Check preconditions, persist the result
- Make the AWS call (or poll, or verify), persist the request and response verbatim
- Check postconditions, persist
- Mark the step `done` / `failed`, persist
- Decide the next step from the plan

All action-specific behavior lives in the *plan*, not the worker. New action types add new plan compilers, never new worker code.

**Why this matters:** there is exactly one code path that touches AWS. Auditing "what could possibly happen" reduces to reading one ~200-line file plus the plan.

### 3. Verbatim capture of every AWS call

For every `aws_api_call` step, the worker writes an audit event containing:

- The full request (service, operation, params), redacted only for known-secret fields
- The full response or error, verbatim
- The AWS request ID (correlation key into AWS CloudTrail)
- Wall-clock start, end, and duration
- The IAM identity used (role ARN, session name, session policy hash)
- The hash-chain link

The AWS request ID is load-bearing. It lets anyone cross-reference Casper's audit log against AWS's own CloudTrail and confirm independently that the action happened, that it happened with the claimed credentials, and that nothing else happened from those credentials. Casper's audit is locally complete; CloudTrail is the external witness.

### 4. State persists between every step

After every step completes or fails, the worker commits a single transaction containing:

- The step result
- The new state-machine state
- The audit events generated by that step
- The next step's `pending` row

If the process crashes, restart resumes from the last committed step — never re-executing a completed AWS call, never skipping one.

Mechanism: a `step_executions` table with `(plan_id, step_id, status, started_at, finished_at, request, response, error)`. The worker uses `SELECT … FOR UPDATE SKIP LOCKED` to claim the next `pending` step. Idempotency keys on AWS calls (where the API supports them) prevent double-execution if a crash happens mid-call.

**Why this matters:** there is no in-memory state that matters. The DB is the truth. You can stop the worker, inspect the DB, and know exactly where things are.

### 5. Live view and audit log are the same data

The UI's live execution view and the historical audit log read from the same table. Live view is `SELECT … WHERE plan_id = ? ORDER BY at` with a Postgres `LISTEN/NOTIFY` push to the browser (or a 1s poll for v1). When execution completes, the same rows are the audit log.

There is no "log to one place for live, another for history" split. The data is identical because it *is* the same data.

This is the single biggest lever for transparency: the user is watching the audit log being written, in real time, as the action runs. There is no "what happened behind the scenes." The screen *is* what happened.

### 6. Verification is a step, not a postscript

The verifier is steps in the plan, not a closing routine. For RDS resize:

- Step N: poll `DescribeDBInstances` until `DBInstanceStatus = available` and `DBInstanceClass = target`
- Step N+1: wait `verification_window` (e.g. 5 min)
- Step N+2: query CloudWatch for the success-criteria metric, assert threshold

If any verification step's postcondition fails, the worker triggers the rollback plan automatically. The rollback plan is itself a list of steps; rollback execution generates audit events identically. There is no "rollback mode" — rollback is just running another plan.

**Why this matters:** "the system rolled back" is a claim. The audit log shows the rollback as a sequence of normal steps with normal AWS calls and normal responses. Anyone can verify the rollback actually happened the way it was described.

### 7. Dry-run shares the code path

Before approval, the system runs the plan in `dry_run = true` mode. In dry-run, every `aws_api_call` write step is replaced with the AWS SDK's `DryRun` parameter (where supported) or a read-only equivalent (`DescribeDBInstances` instead of `ModifyDBInstance`). Polls and verifications run for real (read-only). Predictor results are attached.

The dry-run produces a *preview* audit log — same shape as the real one, marked `dry_run`. The UI shows this before approval. What you see in dry-run is structurally what you'll see during execution; only the write calls differ.

This kills a whole class of "the simulator said one thing, the real run did another" bugs because the simulator and the real run share ~95% of their code.

### 8. Determinism guarantees, in code

Three rules the execution layer enforces in code, not in convention:

- **No side effects outside steps.** The worker may not make AWS calls except via the step interpreter. Enforced by giving the interpreter the only handle to the AWS client; the rest of the worker does not import the AWS SDK.
- **No hidden retries.** AWS SDK retries are disabled (`MaxAttempts = 1`). If a call fails, that's a step failure, recorded as such. Retries become explicit steps in the plan if needed.
- **No clock-dependent decisions inside steps.** Time is passed in. Steps that depend on time (polls, waits) record the time they read. Replays are deterministic given the recorded responses.

## Data model

Four tables underpin the execution layer.

**`plans`** — one row per compiled plan
```
id, proposal_id, kind ("forward" | "rollback"), version,
compiled_at, compiled_from_proposal_hash, steps_json
```

**`step_executions`** — one row per step attempt
```
id, plan_id, step_id, status (pending|running|done|failed|skipped),
started_at, finished_at,
request_json, response_json, error_json,
aws_request_id, identity_session_name, session_policy_hash
```

**`state_transitions`** — proposal-level state machine
```
proposal_id, from_state, to_state, reason, at
```

**`audit_events`** — append-only, hash-chained
```
id, proposal_id, kind, payload_json,
prev_hash, hash, at
```

`hash = sha256(prev_hash || canonical(payload_json))`. Tamper-evident with no extra infrastructure.

## State machine

Per proposal:

```
proposed
  → predicted        (simulator attached blast-radius)
  → policy_checked   (OPA verdict attached)
  → needs_approval | approved | denied
  → planning         (forward plan compiled and persisted)
  → executing        (interpreter running steps)
  → verifying        (post-execution verification steps)
  → done | rolling_back
  → rolled_back | rollback_failed
```

Every transition is a row in `state_transitions` and a row in `audit_events`. The interpreter reads the current state, never assumes it.

---

## Execution flow, end to end

1. Proposal arrives in `proposed` state with the typed action object.
2. Simulator attaches predicted blast radius → `predicted`.
3. Policy engine evaluates → `policy_checked` with verdict.
4. If `needs_approval`, the system waits. Human approves via UI → `approved`. If policy returned `allow` directly, skip to next step.
5. Plan compiler turns the proposal into a forward `ExecutionPlan` and a rollback `ExecutionPlan`. Both persisted → `planning` → `executing`.
6. Interpreter walks the forward plan step by step. Each step: claim, run preconditions, execute, capture, run postconditions, commit. Audit events written through the entire walk.
7. After the last forward step, state moves to `verifying` and the verification steps run. (These are part of the same plan; the state transition is for human-readable clarity.)
8. If verification passes → `done`.
9. If verification fails (or any step fails with `on_failure: rollback`) → `rolling_back`. The rollback plan is executed by the same interpreter. Outcome → `rolled_back` or `rollback_failed`.
10. Throughout, the UI subscribes to `audit_events` for the proposal and renders the live timeline. After completion, the same rows are the permanent audit record.

---

## What the user sees

A vertical timeline. One row per step. Each row shows:

- Step description
- Status (`pending` → `running` → `done` / `failed`)
- Duration
- Expandable to show the full request and response

The timeline updates live during execution. When the action completes, the timeline *is* the audit log. No refresh, no "view history" button — the screen the operator just watched is the permanent record.

That UI plus the dry-run preview *is* the transparency story. The user sees the plan before approving, watches it execute step by step against real AWS, and ends up with a complete record they can re-read or share.

## Build order

Block 1 of the project — implement in this order:

1. Define `ExecutionPlan` and `Step` types in Go.
2. Write the plan compiler for the first action: `RDSResizeProposal → ExecutionPlan`.
3. Write the step interpreter — the only code that touches AWS.
4. Persistence: `plans`, `step_executions`, `state_transitions`, `audit_events`.
5. Wire the live-update endpoint (`LISTEN/NOTIFY` → SSE to the UI).
6. Build the timeline UI on top.
7. Add dry-run mode — a flag through the interpreter, not a separate code path.

The execution layer is "done" when, looking only at the audit log, anyone can answer:

- What was proposed?
- What was the plan?
- Which credentials executed it?
- What did each AWS call return, verbatim?
- Did verification pass? On what data?
- If it rolled back, what did the rollback do?
- Has the log been tampered with? (hash-chain check)

If a future user — or auditor, or you, six months later — can answer all of those without reading the codebase, the execution layer is transparent.
